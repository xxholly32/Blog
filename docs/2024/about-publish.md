# hido 前端构建失败问题的思考

## 背景

最近一直会存在构建hido构建失败的情况，而且每次都是不同的原因，我分析了近期200份构建数据，其中包含构建问题的有 154 个；其中前端构建报错的有 70，占比45.5%；其中npm sha 报错的有，37个，占前端构建占比52.9%；

我们组也整理过具体和前端相关的问题与原因，这个可能会帮助到一些同事，内网链接无法访问

但我还是想针对深层的原因进行一些思考，希望能够找到一些解决方案。

## 构建问题分析

### 1. 环境问题

在我以前的经验中有，svn地址写错，用户名密码设置错，（杭州和成都的svn路径不同）；构建命令写错，构建命令中的参数写错；环境不同导致一些命令指令不同，比如设置nodejs版本；补丁和基线的构建方式不同；

> 这里还有一个比较大的就是 af 仓库不稳定的问题，这个我想不在本次讨论范围内，后面也不会做赘述

### 2. 前端部分

#### 2.1 构建指令无法调试

我们部门一般把构建指令，写在 pom.xml 文件里面，这个文件前后端都能操作，一般都是从其他组件复制过来的。这里有一个很大的问题是：前后端是分离的，前端本地没有 java 环境，后端本地没有 nodejs 环境，不知道修改了是否成功了，必须提交代码去走一次 hido 构建才可以。

#### 2.2 nodejs 版本问题

据我了解，hido 的构建环境中，nodejs 版本是 10，而最新的是20 ，nodejs官网持续维护的也是 18。而我们这里最高的是 18。

一般基线的话后端都会去设置 nodejs 的版本号的，但定制和补丁不同，用的默认都是 10；这个就会存在严重的问题。甚至一些连锁反应。当然hido 的默认值改成 16，18 也不一定是正确的。

#### 2.3 Jenkins 环境问题

我们的构建应该是使用 Jenkins，具体我没有深入研究过，但是不同的环境，服务器，里面肯定是有多个任务在跑的，那共享一个环境肯定会存在问题，比如有个组件 执行了 `npm -g instal npm` 一些奇怪的操作，导致环境不稳定。

#### 2.4 npm sha 报错

报错信息：

```bash
npm ERR! code EINTEGRITY
npm ERR! sha1-P17WZYPM1vQAtaANtvfoYTY+OI8= integrity checksum failed when using sha1: wanted sha1-P17WZYPM1vQAtaANtvfoYTY+OI8= but got sha512-+ux3lx6peh0BpvY0JebGyZoiR4D+oYzdPZMKJwkZ+sFkNJzpL7tXc/wehS49gUAxg3tmMHPHZkA8JU2rhhgDHw== sha1-Z79X9HWPAu3oj7KhcS/vTRU1i+M=
```

大部分的都是 fsevents 这个包的 sha 值不同导致，在不同的nodejs环境下的情况是不一样的。这个在高版本的 nodejs 环境下出现概率较低，具体的规律我没有摸得很清楚。如果有清楚的同事可以私信我一下。

综上所述，其实大部分的问题还是环境问题，可以说是 Jenkins 环境问题，hido 做了很多的工作，比如让我们去设置nodejs版本，java版本，清空缓存等。无法进行本地构建测试。

所以我个人考虑的还是从容器编排的角度去思考，并简化或者解耦一些 hido 的构建操作，把能力回归到本地开发者手中。

## docker 容器编排的尝试

我个人将 ISC 2.1.1 里面的 门禁，视频，停车场，消费，组件拿出来，改造成了容器编排，并输出了个人的最佳实践，我先说下结果。

> 每个组件测试了 7~10 次

构建所有的内容在这个文档内：hido 构建数据统计

### 采用容器编排后的结果

+ 容器编排一旦成功，就基本上是成功的，之前的编译模式有可能还存在编译错误的情况（特别是nodejs版本较低的版本）
+ 容器编排，可以支持本地编排，效果和 hido 构建一致
+ 容器编排可以减少一些 hido 的环境配置
+ 容器编排的时间这块不太好统计，总体上我看了下和之前的编译模式不会差特别的多（我个人编写的默认镜像可能还有优化空间）

### 如何本地编排

个人建议在仓库根目录下面建一个 `dockerfile`，内容可以参考我的，拿 pms 举例

```dockerfile
FROM 内网链接无法访问
# 内网链接无法访问

ENV OA_NAME xiangxiao3
# 注意信息安全
ENV OA_PASSWORD xxxxxxxxxx
ENV GIT_BRANCH test_jsd

WORKDIR /usr/local/code

RUN  git clone -b ${GIT_BRANCH} 内网链接无法访问

WORKDIR /usr/local/code/pms

# 这里以后对应的是 hido 的构建
RUN mvn package -Dmaven.test.skip=true
```

然后在根目录下面执行

```bash
docker build -t pms .
```

如果输出都 SUCCESS 的就说明成功咯，本地就这么调试，是不是非常方便。前端后端都能做。

![](/imgs/docker-build.png){data-zoomable}

> 也不会出现类似，`为什么你电脑不行，我电脑可以的` 甩锅操作了

### hido 中如何使用

在 `hido` 这里选容器编排，然后在构建的时候，选择对应的镜像即可。

![](/imgs/hido-docker.png){data-zoomable}

注意这里的下拉框其实是可以输入的。就先你想要的镜像名称，然后点击保存，就可以了。

在构建命令里面的话，你 dockerfile 里面 `git clone` 后执行什么就写什么。下载代码不用去做，hido会把代码帮我们下载好，你只需要执行脚本即可。

> 这里只是构建，没有加入移动成果物，还有 `wrappingtool` 的一些操作，这个后续有其他同事会输出

然后点构建就 ok 了

## 总结

容器编排主要解决的还是说前端编译偶尔会出错，以及无法支持本地构建。以及保证在未来的时间点都不会出现问题，或者问题可控。不用把解释权交在 hido 手中。有兴趣的同事可以尝试一下。

后续容器编排肯定会成为趋势，对于基础镜像公司也会统一有人去管理，现阶段还是预研期；对于我个人发布的基础镜像请不要覆盖，有优化方案的可以 hiklink 给我，或者提 issues
