# 使用 Cursor 做多语言

## 背景

最近有个项目要做国际化，比较赶，而且一些基线项目我们用 `Vue 3` 重写以后，感觉还得做多语言，我就想着预研一下吧；之前我们组也做过一些工具去自动处理，比如 i18n-generator；但翻译的话适配 `Vue` 的语法可能较差，因为有 `Vue 2`、`Vue 3` 的项目，而且现在生成的多语言都是拼音的首字母，感觉缺少了语义化。

那我想着，用用 `Cursor` 试试，刚好 `Cursor` 的设计总监 **Ryo Lu** 分享了 [Cursor 的 12个使用技巧](/blog/2025/cursor-12-tips.html)，拿来套用下。

## 翻译文件的插曲

我们其实有 **2个多语言文件**，中文和英文么，然后中文文件和英文文件的 `key` 有一些不一样，我需要先把英文的 `key` 补充

> 因为我们多数情况下在中文环境下开发，那英文的 `key` 就会缺失很多

那2个文件本身就有很大的差异，那我就直接把中文翻译文档的内容放到英文翻译文档中，然后让 `Cursor` 帮我翻译

由于我的文件很大，里面的内容大概 **8000多条**

我先用的是 `Claude-4` 的 `Sonnet` 模型，它一次性会读大概 **20行** 数据吧，并且帮我翻译了一下；但我发现呢，经过了几轮自己的翻译，它就开始打算写 `Python` 脚本，然后自顾自的提炼一些通用的中文-英文的通用词汇翻译。但这个提炼毕竟不是我想要的，我需要的是还是一个一个的精准翻译。

![](/imgs/translation-with-cursor/1.png)

我又用了 `Gemini-2.5-Pro` 模型，感觉也是一样，但我感觉并没有说比 `Claude` 好的地方，而且几轮翻译下来后面它还断了，无法继续。

最后我还是用 `Cursor Tab` 的方式去翻译，但 `Tab` 一次性只能翻译 **4行** 文件（而且 `Cursor` 无法配置这个）。最终折腾了 **2个多小时**，搞定。其中我发现中文翻译的 **6000多行** 都是设备或者事件的中文，这块其实不应该在前端存。我就直接删了，所以最终我大概按了 **400多次** `Tab`，小拇指都按抽筋了。

--- 

这里其实就能看出 `Cursor` 的局限，或者 `Cursor` 本身设计就是如此，他并不是说让能够一下子帮你解决所有问题，需要一点一点精确的去帮你搞定，这样的准确度，可能是最高的。当然它也会偷懒，发现一些规律后会自顾自的开始做一些脚本，也不问你同意不同意~

## 开始翻译

### 总体设计

就是参考了 `Cursor` 使用技巧的 **第7条** ，我将我的设计文档放到了项目根目录的 `/.cursor/rules` 中，建了一个 i18n-rule.mdc 的文件

那具体文件内容如下：

```md
# 多语言规则

本项目基于 `vue-i18n` 实现多语言支持。

## 规则说明

1. **所有中文内容**（包括 `template` 和 `script` 中的中文）都必须使用 `t('key')` 进行包裹，禁止直接写死中文。
2. 多语言文件位置：
   - 中文翻译文件：`public/static/i18n/zh_CN/translate.json`
   - 英文翻译文件：`public/static/i18n/en_US/translate.json`
3. 多语言 `key` 命名规则：
   - 必须以 `visitor` 开头
   - 必须以 `_name` 结尾
   - 示例：`visitor.i18n.welcome_name`
4. 当遇到新的中文内容时：
   - 首先在 `translate.json` 中查找对应的 `key`
   - 如果已存在 `key`，则直接复用
   - 如果不存在，则在 `index.json` 中新增 `key`，并将中文内容作为 `value`
5. 所有多语言 `key` 必须语义化，便于维护和复用
6. 组件/页面新增时，需同步补充对应的多语言 `key` 到对应的多语言文件中
7. 如果需要翻译的中文在 **中文翻译文件** `public/static/i18n/zh_CN/translate.json` 中已经有了，那尽量使用已有的多语言 `key`
8. 不需要检查和处理 `Vue` 文件的格式是否正确
9. 不需要检查和处理多语言 `JSON` 文件的格式是否正确

## 示例
```vue
<template>
  <div>{{ $t('visitor.i18n.welcome_name') }}</div>
</template>

<script>
export default {
  methods:{
    test () {
      console.log(this.$t('visitor.i18n.welcome_name'))
    }
  }
}
</script>
```

当然这个文案也是我用 `Cursor` 的 `Ask` 模式让 `Cursor` 帮我写的，然后我再稍微修改下。

我是一个菜单一个菜单翻译的，不能全局去问，因为 `Cursor` 的上下文有限；由于有了我的 `prompt` 就很简单艾特几个文件以后，就说帮我将这些文件翻译成英文；

它的步骤大致都是这样的：

1. **开始确定一个文件**，找里面的中文
2. **将文件里面要翻译的中文提取出来**
3. **将多语言的 `key` 放到中文的 `translate.json` 中**，校验格式是否正确
4. **将多语言的 `key` 和英文翻译放到英文的 `translate.json` 中**，校验格式是否正确
5. **然后修改文件，并处理硬编码问题**
6. **重复 1-5 步，做出总结**

上面第二张这个图片，里面面有一个 **确认文件的按钮**，它会改多个文件，你可以一个文件一个文件去审查，如果它翻译错误的，你可以回滚它的改动。其实我要做的就是提出我的问题，然后大概等一会它执行，然后确认它的内容就行了。

除了上面一点外，我个人觉得比较好的点：

1. **Vue 文件的准确率非常高** `95%` 以上，基本不会报错，一些复杂情况它也能处理到位（这个是最让我觉得最好用的点）
2. **翻译的英文格式不仅符合要求**，且还根据菜单会自己加入中间的段，而且英文 `key` 通俗易懂
3. **有总结**，你会找到可能有些文件它没翻译，那你可以根据总结有需要的浏览下（虽然这个比较耗时，但我觉得这个是值得的）

**会存在的一些问题**：

1. **文件校验问题**：当它生成文件以后，它会对文件做校验，每次在 `JSON` 或者 `Vue` 文件加内容后，如果编辑器校验不通过，它会去修复

   这里应该是我编辑器和配置的问题，`JSON` 文件默认都会有错误，那它都会检测下，但每次它都会发现其实没问题；但这个会比较耗时，我在 `prompt` 加了也没有用，不过 `Cursor` 也有这样的配置，默认是开启的。

   其实它对 `Vue` 文件也会校验，去自动处理我的代码格式问题。有 **1次** 它处理错误了（不过其实是我们代码的问题）。

   这块我建议还是尽量在翻译前全局处理下处理的文件前，文件侧边尽量不要小红点；

2. **重复 `key` 问题**：它不会去处理重复的多语言 `key`，虽然它偶尔会执行这个步骤，但我发现还是有很多地方都是有重复的中文。可能是我的 `prompt` 问题；

3. **文件处理限制**：能处理的文件比较有限，我一般最多不超过 **5个文件** 吧，但它还是偶尔会漏掉几个。翻译也会偶尔漏掉几个。

4. **语法错误**：会存在一些错误，比如有 **1次** 会在 `template` 中写了 `this.$t('')`，它就会报错，然后我手动去改了下。

## 翻译数据统计

没有完整的数据吧，只有一个大概的：
- 帮我新增 **800多条** 英文多语言 `key`
- 处理文件 **40+**
- 修改有效代码 **1000+**
- 翻译我基本只花了 **3个小时**，而且大部分是等待他回复的时间，如果不用 `Cursor` 的话，我估计得花个 **3天** 吧。

> 而且我在 `Cursor` 的 `token` 使用数上，其实并没有开发代码用的多。（`Cursor` 好像只统计 `Tab` 的使用次数）

## 总结

我把我的翻译的规则放到了 这里，有需要的可以参考或者魔改下。

对于 [Cursor 的 12 个使用技巧](/blog/2025/cursor-12-tips.html)，我觉得大部分还都是有用的，其实看这个就够了。

**需要强调的是**，尽量不要想着将很复杂的任务交给 `AI`，还是需要拆解和逐步指导 `AI`，才能更好的和 `AI` 共同协作。
